# Retail BI Pipeline — audit-first подготовка данных для Yandex DataLens

Учебный проект по курсу визуализации данных (ИАВД-11, МФТИ). Берёт сырую
выгрузку транзакций онлайн-ритейлера (`Retail.xlsx`, ~947 тыс. строк за
декабрь 2009 – октябрь 2011), проводит её через аудит-ориентированный
пайплайн и формирует готовую звёздную схему для BI-дашборда в
**Yandex DataLens**.

Главная идея — **максимум подготовки в Python, минимум логики в BI**.
Все агрегаты, флаги, категории и справочные поля рассчитываются заранее,
чтобы на стороне DataLens оставалась только визуализация.

## Что делает проект

1. **Загрузка** `Retail.xlsx` (один лист `online_retail_data`).
2. **Фильтрация служебного слоя** — поле `rnd = True` помечает строки,
   которые не являются частью реальных бизнес-операций (нулевые цены,
   пустые описания, технические записи). Они исключаются до нормализации.
3. **Нормализация** текстовых полей: `StockCode`, `Description`, `Country`,
   `Channel`, `Customer ID`. Устраняется регистровая рассинхронизация
   (`47566B` vs `47566b`), лишние пробелы, нестандартные названия стран
   (`EIRE` → `Ireland`, `RSA` → `South Africa`).
4. **Дедупликация** — сначала полные дубликаты строк, затем дубликаты по
   бизнес-ключу.
5. **Классификация** каждой строки в один из 11 типов (`line_type`):
   `sale`, `return`, `cancelled_sale`, `shipping`, `discount`,
   `manual_adjustment`, `commission_fee`, `bad_debt`, `gift_voucher`,
   `test`, `unknown`. Для каждого типа зафиксирован приоритет правил.
6. **Реклассификация same-day cancellations** — продажи и возвраты,
   оформленные в один день по одному клиенту и товару, помечаются как
   `cancelled_sale` и выводятся из основных метрик.
7. **Построение измерений** (`dim_*`): товары, клиенты, календарь,
   страны, категорийные метрики возвратов.
8. **Построение фактов** (`fact_*`): три гранулярных таблицы операций
   + агрегированная дневная сводка для быстрой аналитики.
9. **Экспорт** в три формата: parquet (основа), CSV (ручная проверка
   и загрузка в DataLens) и единый Excel-workbook (`retail_datalens_export.xlsx`)
   со всеми витринами на отдельных листах.
10. **QA-артефакты** — 17 таблиц аудита (дубликаты, возвраты, экстремумы,
    пропуски, маппинг стран и т.д.) и JSON-сводка запуска в `reports/`.

## Исходные данные

В репозитории **нет** `Retail.xlsx` (файл около 80 МБ, не укладывается
в разумные размеры git-репозитория). Для воспроизведения пайплайна его
нужно положить самостоятельно.

**Куда положить:**

- рекомендуемый путь: `data/raw/Retail.xlsx`
- альтернативно поддерживается: `Retail.xlsx` в корне проекта

Пайплайн ищет файл сначала в `data/raw/`, затем в корне. Имя листа —
`online_retail_data` (задаётся в `src/config.py`).

**Что внутри:** транзакционные строки онлайн-ритейлера подарков и товаров
для дома из Великобритании. Ключевые поля: `Invoice`, `StockCode`,
`Description`, `Quantity`, `Price`, `InvoiceDate`, `Customer ID`,
`Country`, `Channel`, `rnd`. Последнее поле — служебный флаг: все
строки с `rnd = True` фильтруются до нормализации (это аномалии
источника, а не реальные операции; расследование этого поля — раздел 11
в notebook).

## Подготовка окружения

Проект использует менеджер пакетов **[uv](https://docs.astral.sh/uv/)**.
Под Windows есть готовые bat-обёртки для всех операций.

**1. Установить `uv`** (если ещё не установлен):

```bat
uv-install.bat
```

**2. Создать виртуальное окружение и установить зависимости:**

```bat
uv-make.bat
```

Эта команда создаёт `.venv/` и ставит все пакеты из `pyproject.toml`
(pandas, numpy, openpyxl, pyarrow, jupyter).

**3. Для полной очистки** (удалить `.venv` и временные файлы):

```bat
uv-clean.bat
```

**Без Windows-батников.** Всё то же самое вручную:

```bash
uv sync                      # или: python -m venv .venv && pip install -e .
```

## Запуск пайплайна

**Одной командой через bat-обёртку:**

```bat
uv-run.bat
```

Эквивалент:

```bat
uv run python -m src.pipeline
```

**Что происходит:**

- чтение `Retail.xlsx` (около 1 минуты на SSD)
- фильтрация `rnd = True`, нормализация, классификация, реклассификация
- построение измерений и фактов
- сборка QA-артефактов
- экспорт parquet/csv во все целевые каталоги
- запись Excel-workbook `retail_datalens_export.xlsx`
- запись JSON-сводки в `reports/pipeline_run_summary.json`

Полный прогон на типичной машине занимает 4–5 минут (основное время —
чтение Excel и запись Excel-workbook).

## Что появляется на выходе

**`data/processed/`** — витрины для DataLens:

| Файл | Описание |
|---|---|
| `fact_sales_lines` | Продажи: одна строка = одна позиция инвойса |
| `fact_return_lines` | Возвраты: аналогично, с отрицательными суммами |
| `fact_service_lines` | Доставка, скидки, отменённые продажи, корректировки |
| `fact_daily_summary` | Дневная сводка в разрезе страна × канал × категория |
| `dim_product` | Справочник товаров с канонической категорией |
| `dim_customer` | Клиенты с LTV-сегментом, флагом возвратов, основной страной |
| `dim_date` | Календарное измерение (год, квартал, месяц, флаг неполного месяца) |
| `dim_country` | Страны с региональной группировкой и флагом UK |
| `dim_category_metrics` | Return rate и объёмы по товарным категориям |

Каждая витрина сохраняется в трёх форматах:
`.parquet` (основа), `.csv` (с BOM `utf-8-sig`, для ручной проверки),
и как лист в `retail_datalens_export.xlsx`.

**`data/qa/`** — 17 аудитных таблиц: профиль raw-слоя, пропуски,
полные и бизнес-дубликаты, кандидаты в возвраты, bad debt, анонимные
транзакции, экстремумы, распределение по `line_type`, сверка слоёв
raw → processed и другие.

**`reports/pipeline_run_summary.json`** — компактная сводка запуска:
число строк на каждом слое, распределение по `line_type`, размеры
витрин. Используется для быстрой проверки «пайплайн прогнан, числа
сошлись».

## Notebook: пошаговый аудит

Главный живой документ проекта — **`retail_audit_and_pipeline.ipynb`**.
Это не отчёт после факта, а последовательный аудит данных:

1. Загрузка и профилирование raw-слоя
2. Расследование поля `rnd`
3. Дубликаты: технические vs бизнес-ключ
4. Возвраты и бухгалтерские корректировки (bad debt, Amazon fees)
5. Нулевые цены и служебные коды (shipping, discount, gift)
6. 214 тысяч анонимных транзакций — отдельный сегмент
7. Качество справочника товаров (код → описание)
8. Экстремумы и полнота последнего месяца
9. Итоговая классификация `line_type`
10. Итог расследования `rnd` и решение об исключении
11. Сборка витрин через `run_pipeline()` и превью результатов
12. Описание модели данных и связей для DataLens
13. Портрет бизнеса по итогам очистки

Каждый шаг — это **наблюдение → вывод → решение** с цифрами. Логика
трансформации не дублируется в notebook — это вызовы функций из
`src/`, благодаря чему notebook остаётся читаемым.

## Модель данных (звёздная схема)

Три фактовые таблицы разделены по типу операции. Это принципиальное
решение: метрики для продаж, возвратов и сервисных строк считаются
по-разному, и смешивать их в одну таблицу означает усложнять каждый
чарт в BI.

```text
              ┌──────────────┐
              │  dim_date    │
              └──────┬───────┘
                     │
┌──────────────┐     │    ┌──────────────┐
│ dim_product  │─────┼────│ dim_country  │
└──────┬───────┘     │    └──────┬───────┘
       │             │           │
       │      ┌──────▼──────┐    │
       └──────│ fact_sales  │────┤
              │ fact_return │    │
              │ fact_daily_ │    │
              │  summary    │    │
              └──────┬──────┘    │
                     │           │
              ┌──────▼───────┐   │
              │ dim_customer │───┘
              └──────────────┘
```

**Ключи связи:**

- `dim_date.date` ↔ `fact_*.invoice_date`
- `dim_product.stock_code_norm` ↔ `fact_*.stock_code_norm`
- `dim_country.country_name` ↔ `fact_*.country_name`
- `dim_customer.customer_id` ↔ `fact_*.customer_id`

Дополнительно в факты добавлены raw-поля для кросс-фильтрации в
DataLens (без них алиасы чартов не пробрасывают фильтры):
`year_month` (строка `'YYYY-MM'`), `region` (lookup из `dim_country`),
`is_uk` (`0/1`).

## Ключевые решения по очистке данных

Кратко о том, **почему** пайплайн устроен именно так. Детальное
обоснование каждого решения — в notebook, раздел соответствующего шага.

- **`rnd = True` → отфильтровано до нормализации.** Это служебный слой
  (100% аномалий источника: нулевые цены, пустые описания, отрицательные
  цены). 37 213 строк.
- **Дедупликация до реклассификации.** Асимметричные дубликаты в парах
  sale/return нарушили бы подсчёт `n_sales == n_returns` и часть
  реальных same-day cancellations была бы пропущена.
- **Same-day cancellations → отдельный класс.** Продажа и возврат того
  же товара тем же клиентом в тот же день — это не две реальные
  операции, а отменённый заказ. Попадают в `fact_service_lines` с типом
  `cancelled_sale` и исключаются из основных метрик выручки.
- **Анонимные клиенты не участвуют в матчинге пар.** У всех анонимов
  `customer_id = 'ANONYMOUS'`, и при группировке они объединялись бы в
  одного «супер-клиента». Один покупатель купил, другой вернул тот же
  товар в тот же день — это не отмена.
- **Булевы поля в CSV как `0/1`, а не `True/False`.** DataLens работает
  поверх ClickHouse, где `Boolean = UInt8`. Текстовый `True/False`
  парсится нестабильно на больших таблицах, `0/1` — надёжно.
- **Денежные поля округляются до 2 знаков.** Плюс `float_format='%.2f'`
  при записи в CSV — страховка от float-артефактов вида `1.5699999999999998`.
- **Даты без времени.** В источнике `InvoiceDate` всегда содержит
  `00:00:00`, время не несёт смысловой нагрузки. В витринах — `date`.
- **Категоризация товаров по ключевым словам.** Порядок категорий в
  `config.product_category_keywords` задаёт приоритет: сначала
  специфичные (Lighting, Christmas), затем общие (Home Decor, Kitchen).
  Это нужно, чтобы `T-LIGHT HOLDER` ушёл в Lighting, а не в Home Decor
  из-за слова `HOLDER`.
